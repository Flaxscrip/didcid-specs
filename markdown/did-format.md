## DID Format

[[def: did:cid, A DID method that uses Content Identifier (CID) v1 hashes as the method-specific identifier, anchored to IPFS for creation and a pluggable registry for updates]]

The `did:cid` method conforms to the [[ref: DID-CORE]] generic DID syntax. DIDs using this method have the following ABNF:

```
did-cid        = "did:cid:" cid-identifier
cid-identifier = CIDv1-base32

; CIDv1-base32: a CID v1 encoded in standard base32
; Example: bafkreiawdmk6fmqc5p237vffyctazpzdgvgqfdj2i3hx2idtodxkwhyj5m
```

The `cid-identifier` is the IPFS content identifier generated by applying [[def: JCS, JSON Canonicalization Scheme (RFC 8785) — a deterministic encoding for JSON that ensures consistent hash computation across implementations]] to the initial [[def: seed document, The first operation object (type "create") that is deterministically encoded via JCS and pinned to IPFS, whose CID becomes the DID suffix]] and computing its CID v1 in standard base32 encoding.

### Example

`did:cid:bafkreiawdmk6fmqc5p237vffyctazpzdgvgqfdj2i3hx2idtodxkwhyj5m`

### DID URL Fragment Support

[[ref: DID-CORE]] defines DID URLs that may include path, query, and fragment components. The `did:cid` method supports the fragment component (`#`) for referencing verification methods within a DID document:

`did:cid:bafkreiawdmk6fmqc5p237vffyctazpzdgvgqfdj2i3hx2idtodxkwhyj5m#key-1`

::: note
DID URL path, query, and service parameter dereferencing are defined in [[ref: DID-CORE]] but are not currently implemented by the reference Gatekeeper. Implementations SHOULD treat unrecognized DID URL components according to the DID Core error handling specification.
:::

### Method-Specific Identifier

The method-specific identifier is a CID v1 in base32 encoding. This identifier is **deterministic** — the same seed document always produces the same CID — which enables [[def: content-addressed identity, A property of did:cid where the DID suffix is derived from the hash of the seed document, making the identifier self-certifying and independently verifiable]].

::: warning
Because the DID suffix is derived from the content hash of the seed document, any change to the seed document produces a *different* DID. There is no mechanism to alter a seed document after creation — mutations are expressed through successive update operations recorded on the registry.
:::
